Актуальность связана с накоплением ссылок на внешние ресурсы в учебных и рабочих заметках: при базовом Markdown и HTML без специализированного контура пользователь дублирует заголовок, описание и превью вручную, а клиентский код карт и виджетов конфликтует с серверным рендерингом. Цель — спроектировать и описать фрагмент персональной веб-платформы с MDX, компонентами карточек и серверным извлечением метаданных Open Graph и JSON-LD во внутреннюю модель. Задачи: разделить серверный HTTP-агент с кэшем и клиентский редактор, задать контракт API и поведение при ошибках сети. Результат: контур «редактор — сохранение — карточка», инварианты SSR, приоритеты полей метаданных, направления развития источников, офлайна и автодополнения без обязательной низкоуровневой правки разметки. Практическая значимость — снижение трудозатрат на оформление ссылок и повышение воспроизводимости архитектурного решения в учебных и пилотных внедрениях.

Ключевые слова: веб-платформа, MDX, метаданные, Open Graph, JSON-LD, SSR, карточка ссылки.

Введение

Форматы Markdown и HTML задают компромисс между удобством записи и выразительностью: чистый Markdown плохо переносит интерактив и повторно используемые блоки; HTML вручную в заметках применяют реже. MDX допускает JSX-компоненты в потоке Markdown-текста; компиляция строится на цепочке парсеров (семейство Micromark) и выдаёт код для выбранного рантайма (React). Карточка ссылки как компонент унифицирует хранение: в документе остаётся URL и при необходимости закэшированные поля title, description, image.

Прямой HTTP-запрос к стороннему origin из браузера упирается в CORS; извлечение og:* и JSON-LD выполняют на сервере или через контролируемый прокси. Отдельная задача — не подключать на сервере пакеты, тянущие window/document (типичный случай — обёртки картографических библиотек), иначе SSR падает или требует условной загрузки. Работа фиксирует архитектурное разделение: данные и метаданные — на сервере; редактор, карта, интерактив — на клиенте с динамическим import.

Исследование

1. Постановка задачи и функциональный контур

Требуется единое окно редактирования, вставка URL статьи или API-документации, автоматическое заполнение карточки полями из метаданных, сохранение документа без обязательного «сырого» HTML. Нефункциональные требования: предсказуемый SSR, модульность слоя метаданных и UI, возможность расширения списка извлекателей (OG, JSON-LD, резерв по <title> и meta description).

2. Извлечение и нормализация метаданных

HTTP-агент запрашивает ресурс, парсит HTML (дерево DOM или потоковый разбор). Для Open Graph извлекают элементы meta с атрибутом property в пространстве имён og:title, og:description, og:image, og:url; для JSON-LD — встроенные блоки application/ld+json с последующим разбором JSON и сопоставлением типов Schema.org (WebPage, Article и др.) к внутренней DTO. Конфликты полей разрешают приоритетом: og:title над устаревшим twitter:title при согласованной политике; при отсутствии изображения — плейсхолдер или пропуск блока изображения. Кэш ответа (ключ — нормализованный URL, TTL — параметр развёртывания) снижает частоту обращений к внешним хостам и стабилизирует задержку редактора. Нормализация URL перед запросом (схема по умолчанию https, удаление фрагмента #…) снижает дубли кэша.

3. MDX-редактор и автодополнение

Редактор опирается на модель «исходный MDX + превью». Автодополнение строится на анализе контекста курсора: внутри JSX — имена локальных компонентов и пропсы; в тексте Markdown — вставка шаблонов списков и ссылок. Для карточки триггером служит ввод URL в компонент или сочетание клавиш «вставить ссылку как карточку»; после debounce выполняется запрос к API метаданных, результат маппится в пропсы компонента. Ошибки сети и таймауты не должны ломать документ: сохраняется URL, карточка рендерится в деградированном виде (домен + текст ссылки).

4. Клиент–сервер и изоляция Leaflet

Серверный маршрут отдаёт HTML-оболочку и данные; компоненты с обращением к window подключают через dynamic(..., { ssr: false }) либо аналог в используемом фреймворке. Карта и прочие виджеты с прямой работой с DOM остаются в клиентском бандле. API метаданных возвращает JSON фиксированной схемы (поля title, description, imageUrl, canonicalUrl, source); единый формат упрощает кэш и тестирование. Пакетный анализ зависимостей (граф импортов) на этапе сборки выявляет случайное попадание браузерных модулей в серверный чанк до выката.

5. Сравнение с базовым Markdown-редактором

В классическом Markdown ссылка остаётся строкой; плагины превью подключают разрозненно. Предлагаемый контур объединяет превью, MDX и агрегацию в одном конвейере. Ручной перенос заголовка и картинки с сайта источника исключается для типовых страниц с корректными og-тегами или JSON-LD.

6. Ограничения и риски

Часть сайтов блокирует нестандартный User-Agent или отдаёт пустой HTML ботам; часть — только после исполнения JavaScript: без headless-браузера метаданные не извлекаются. Юридический контур: соблюдение robots.txt, ограничение частоты запросов к одному хосту, политика хранения изображений (ссылка против локальной копии). Логирование URL может затрагивать персональные данные при запросах к персональным страницам — требуется минимизация логов и срок хранения.

7. Контракт API метаданных и обработка ошибок

Конечная точка возвращает код 200 и тело JSON при успехе; 400 — некорректный или не-http(s) URL; 404 — ресурс недоступен; 502 — таймаут или сетевой сбой при обращении к источнику. Поле error в JSON при неполном успехе позволяет клиенту показать карточку в деградированном режиме без повторной отправки всего документа. Заголовки Content-Security-Policy и ограничение размера ответа HTML на стороне агента снижают риск «тяжёлых» страниц и вложенных редиректов; число редиректов ограничивают константой (например, не более пяти переходов).

Идемпотентность GET-запросов к API метаданных упрощает повторные вызовы из редактора при правках URL. Версионирование схемы DTO (поле schemaVersion) даёт возможность миграции клиента и кэша при расширении набора полей без поломки уже сохранённых документов.

Заключение

Сформирован архитектурный скелет персональной платформы: MDX-документ, серверный слой агрегации метаданных по OG/JSON-LD с DTO и кэшем, клиентский редактор с автодополнением и изоляцией картографических зависимостей от SSR. MVP закрывает цикл «редактор — сохранение — карточка»; дальнейшее развитие — расширение покрытия форматов метаданных, офлайн/PWA, углубление UX автодополнения и политики кэша под нагрузку.

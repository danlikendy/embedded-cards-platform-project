## Порядок слайдов в Figma и логика доклада

**Нумерация слева в файле — это порядок слайдов в презентации; он корректен.** При этом **слайд с основным заключением («спасибо», итоговая мысль) в макете может стоять не последним**: после него идут **дополнительные тематические слайды** (техническая зрелость, сравнение с аналогами, статус и траектория). Это не ошибка вёрстки, а сознательная структура: сначала закрывается **основная линия** повествования, затем — **сжатый блок** для заинтересованной аудитории.

**Как озвучивать:**

1. Пройти слайды **1–9** в обычном темпе (титул, содержание, контекст, проблема, решение, возможности, сценарий, ценность, демо продукта).  
2. На **слайде 10** сформулировать **основное заключение** и поблагодарить за внимание (как на макете).  
3. Явно обозначить переход: *«Дальше у нас есть **дополнительные материалы** по слайдам — по ним пробегусь **кратко**, после этого перейдём к **вопросам**»*.  
4. Слайды **11–13** — **дополнительный блок** (техническая зрелость; отличие от «ещё одного блокнота»; статус и траектория): только ключевые тезисы, без повторного «спасибо» на каждом экране.  
5. После слайда **13** — устный переход: *«На этом по дополнительным материалам всё; готов ответить на ваши вопросы»*.

Веб-версия `index.html` следует этой же последовательности: слайд «Спасибо» идёт **после демо**, затем три слайда с пометкой **«Дополнительные материалы»**.

**Коротко (ориентир по заметке дизайна — держать слайд плотным):**

## Слайд 5. Решение

**Заголовок:** РЕШЕНИЕ  

**Подзаголовок:** Персональная **веб-платформа** (акцент на «веб-платформа»).

**Основной текст (слева / блок):**  
Пользователь пишет в одном окне; карточки внешних источников подтягивают заголовок, описание и изображение.

**Нумерованный список (справа / второй столбец):**

1. Объединяет **MDX** (Markdown + встраиваемые компоненты).  
2. **Агрегацию метаданных** по протоколам вроде Open Graph и JSON-LD.  
3. **Автодополнение** при вводе ссылок и компонентов.

**Нижний акцент (крупно или отдельной строкой):**  
Архитектура рассчитана на **клиентский рендеринг** без утечки Leaflet и DOM-логики на сервер.

## Слайд 6. Ключевые возможности продукта

**Заголовок:** КЛЮЧЕВЫЕ ВОЗМОЖНОСТИ ПРОДУКТА  

1. **MDX-редактор** — привычная разметка плюс компоненты там, где нужна структура или интерактив.  
2. **Встроенные карточки** — ссылка превращается в читаемый блок с превью, а не «голый» URL.  
3. **Агрегаторы метаданных** — единый слой нормализации полей с разных сайтов.  
4. **Автодополнение** — меньше ошибок в путях, быстрее ввод повторяющихся сущностей.  
5. **Клиент-only карта и тяжёлые зависимости** — стабильный SSR, предсказуемое развёртывание.

## Слайд 7. Сценарий

**Заголовок:** СЦЕНАРИЙ  

Пошаговый поток (на макете — стрелки между блоками):

1. Открывает документ в редакторе.  
2. Вставляет ссылку на статью или API — система запрашивает метаданные и показывает карточку.  
3. Добавляет раздел с MDX-компонентом (например, выделенное предупреждение или встроенный виджет).  
4. Сохраняет: всё остаётся в одном потоке контента, без переключения в «режим кода» без нужды.

## Реплики для Q&A

- **Почему MDX, а не только Markdown?** Расширяемость без отказа от читаемого исходника — компоненты в потоке текста.  
- **Что если сайт не отдаёт Open Graph?** Цепочка fallback (JSON-LD, разметка, эвристики по HTML); документ не ломается.  
- **ИИ и конфиденциальность:** уточните политику (локально / API, что уходит в модель).  
- **Запросы к внешним URL:** backend или прокси, кэш, лимиты, политика развёртывания.

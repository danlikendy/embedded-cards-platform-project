# ГЛАВА 3. ПРОЕКТИРОВАНИЕ И РАЗРАБОТКА MVP ПЕРСОНАЛЬНОЙ ВЕБ-ПЛАТФОРМЫ

## 3.1 Архитектура веб-платформы с поддержкой встроенных карточек внешних источников

Два архитектурных ограничения, зафиксированных в п. 2.2, являются формообразующими для всего контура: агрегация метаданных физически невозможна на клиенте из-за политики Same-Origin (T-03), а формат хранения обязан допускать встраивание JSX без проприетарного движка (T-01, T-05). Следствие - неустранимая граница между браузерным SPA и серверным агентом агрегации. Любая архитектура, смешивающая эти слои (например, Obsidian-плагин с прямыми HTTP-запросами к чужим доменам), воспроизводит ограничение, описанное в табл. 7: 40-60% сайтов вернут CORS-ошибку до получения HTML.

Для описания архитектуры применяется модель C4 на уровнях «Контекст» и «Контейнеры». Уровень «Компоненты» детализируется в тех узлах, где граница ответственности неочевидна из диаграммы контейнеров.

На рисунке 11 представлена C4-диаграмма контейнеров MVP. Четыре контейнера разделены по типу хранения и жизненному циклу данных: браузерный SPA (React, CodeMirror 6), API Route агрегации (Node.js, node-html-parser), серверный in-memory кэш (Map с TTL) и хранилище документов (localStorage). Пользователь взаимодействует только с браузерным SPA; серверный контейнер недоступен из браузера сторонних страниц - только через fetch-запрос от того же хоста.

В таблице 13 каждый технологический выбор сопровождён причинно-следственной цепочкой: ограничение → требование → технология.

Слой хранения данных реализован в трёх независимых зонах с принципиально разными жизненными циклами. Документы пользователя сохраняются в localStorage как JSON-массив с полями `id`, `title`, `content` (raw MDX), `createdAt`, `updatedAt`; обновление происходит при каждом keystroke через хук `useDocuments` с debounce на запись. Серверный кэш метаданных - синглтон уровня модуля Node.js: `Map<string, { data: LinkMetadata; expiresAt: number }>`; TTL 24 часа обоснован медианным сроком жизни OG-метаданных публикаций - заголовки и изображения обновляются на порядок реже. Данные внешних сайтов не копируются: платформа хранит только URL в теге компонента и нормализованный DTO в кэше - без изображений, без сырого HTML, без cookies (соответствие ограничению L из PESTLE, табл. 5). Выбор localStorage вместо IndexedDB обоснован синхронным API и достаточным объёмом (5-10 МБ) для текстовых MDX-документов пилота; асинхронный IndexedDB потребовал бы усложнения хука без измеримого прироста для целевого объёма данных.

Контракт API агрегации - единственная точка связи между клиентским и серверным контейнерами. Эндпоинт `GET /api/metadata?url=<encoded>` возвращает `MetadataResponse`. В таблице 14 зафиксированы все поля DTO, их источники и приоритет разрешения конфликтов.

Таблица 14 - Контракт API агрегации: поля DTO и приоритет источников

|------|-----|----------------------|----------------|------------|
| url | string | Запрос | Да | Исходный URL до редиректов |
| canonical | string | og:url → link[rel=canonical] → url | Да | Канонический URL для карточки-ссылки |

Разделение `description` и `excerpt` - ключевая архитектурная точка. `description` содержит OG-текст, написанный авторами сайта в рекламных целях; `excerpt` - фактический контент страницы, извлечённый парсером тела. В большинстве карточек (75% тестовой выборки, § 3.4) `excerpt` информативнее: он отражает реальное начало статьи, а не маркетинговый тизер. Именно это разделение формирует дифференциатор от нативных preview-систем Notion и Craft (табл. 7), где карточка показывает OG-текст.

Однонаправленный поток данных клиентского слоя: `useDocuments` → `MDXEditor` → `useDebounce` → `useMetadata` → `MDXPreview` → `LinkCard`. Состояние не передаётся через prop-drilling глубже двух уровней - это снижает риск регрессий при изменении структуры компонентов и упрощает локализацию ошибок при тестировании.

## 3.2 Разработка MDX-редактора с функцией автодополнения

CodeMirror 6 выбран по трём несводимым причинам. Во-первых, модульная архитектура расширений позволяет добавить обнаружение URL как независимый плагин без вмешательства в Markdown-грамматику - альтернативные редакторы (Quill, Slate) требуют модификации ядра для встраивания нестандартных обработчиков. Во-вторых, библиотека не зависит от JSDOM: это устраняет конфликт с SSR-бандлингом Next.js App Router, который возникает при импорте DOM-зависимых модулей в серверные компоненты (ограничение, зафиксированное в п. 1.3 при анализе стека). В-третьих, CodeMirror 6 обеспечивает производительный рендеринг больших документов через виртуализацию строк - релевантно для документов с большим количеством карточек, где каждая занимает несколько строк MDX.

Редактор инициализируется через `next/dynamic` с `ssr: false`: это обязательное условие, поскольку CodeMirror создаёт DOM-объекты при инициализации, недоступные в серверном контексте. Динамический импорт добавляет задержку первой загрузки (~200 мс), которая компенсируется скелетон-заглушкой.

Функция обнаружения URL встроена в обработчик события `onChange` редактора. При каждом изменении документа регулярное выражение `https?://[^\s"<>)[\]]+` сканирует текст; символы-ограничители `"<>)[\]` выбраны так, чтобы не захватывать URL, уже находящиеся внутри MDX-атрибутов (`url="..."`) или Markdown-ссылок. Найденные URL сравниваются с множеством уже существующих `<LinkCard url="..." />` через второй regex, исключая повторные предложения. Первый незнакомый URL передаётся как `suggestedUrl`.

Debounce-задержка 400 мс выбрана расчётно: NFR T-04 ограничивает суммарное время от ввода URL до отображения карточки значением 3 000 мс. При измеренной медиане ответа API 1 240 мс и P95 2 810 мс (§ 3.4) debounce 400 мс оставляет запас 1 360-1 760 мс для сетевых задержек. Снижение до 200 мс не улучшает UX при вставке из буфера (один клик не требует debounce), но увеличивает число избыточных запросов при наборе URL вручную. Порог 400 мс - компромисс, оптимизированный под сценарий «вставка из буфера обмена», который в сессиях тестирования составил 83% операций добавления ссылок.

На рисунке 12 представлена схема взаимодействия компонентов редактора: от события изменения документа до вставки компонента карточки в MDX-исходник, включая цепочку `onChange → детектор URL → debounce → UrlSuggestion → handleInsertCard`.

Компонент `UrlSuggestion` отображает превью карточки поверх редактора (`position: absolute`, нижний край) до подтверждения вставки. Он самостоятельно вызывает `useMetadata(url)`, что позволяет пользователю оценить результат агрегации прежде, чем текст будет изменён. При подтверждении `handleInsertCard` заменяет голый URL на `<LinkCard url="..." />` в тексте документа; при отклонении URL добавляется в `dismissedUrls` - подсказка не повторяется в рамках текущей сессии. Принцип «редактор не изменяет текст без явного согласия» снижает когнитивную нагрузку и устраняет сценарий «редактор сломал мой документ».

Quick URL bar в боковой панели реализует альтернативный путь, не требующий понимания MDX-синтаксиса. Пользователь вставляет URL в текстовое поле и нажимает Enter; `handleAddUrl` добавляет тег `<LinkCard url="..." />` в конец активного документа. Этот путь специально спроектирован для нетехнической аудитории: не нужно открывать редактор, знать синтаксис компонентов или понимать, что такое JSX. В сессиях тестирования 57% операций добавления ссылок были выполнены именно через Quick URL bar, что подтверждает правильность решения об его размещении на первом экране без модального диалога.

Онбординг первого запуска транслирует ценностное предложение без текстовых объяснений. Полноэкранный оверлей при первом запуске содержит единственное поле ввода; пользователь вставляет URL, получает карточку в реальном времени и видит сравнение форматов: Markdown-ссылка (голый текст) против `<LinkCard />` с заголовком, реальным текстом страницы и обложкой. При нажатии «Добавить в документ» карточка вставляется в активный документ, а флаг `mdx-cards-onboarded-v1` фиксируется в localStorage. Повторный показ исключён. Ключевой UX-эффект: пользователь получает первый результат работы агрегатора ещё до того, как увидит интерфейс редактора.

Пользовательский интерфейс редактора реализует четыре режима отображения, переключаемых через тулбар: «только редактор», «редактор + превью» (50/50), «только превью», «галерея карточек». Режим «галерея карточек» - функциональная особенность, отсутствующая у конкурентов из табл. 4: пользователь видит все карточки документа в виде вертикальной ленты без доступа к коду. Это превращает MDX-документ в read-only curated board, где визуализация контента источников важнее его разметки. Команда palette (`⌘K`) предоставляет быстрый переход между документами, создание нового документа и поиск по содержимому без мыши.

Статус-бар в нижней части отображает счётчики слов, символов, времени чтения и количества карточек в реальном времени. Индикатор «1 URL без карточки» в тулбаре сигнализирует о голых URL, не преобразованных в компоненты - это явный call-to-action, трассируемый к сценарию автодополнения.

Реализованный функциональный объём соответствует MoSCoW-приоритетам табл. 9: Must T-01-T-03 - выполнены; Should T-04-T-05 - выполнены; Could T-06 - выполнен через конфигурацию graceful degradation.

## 3.3 Реализация агрегаторов метаданных и интеграция компонентов платформы

Серверный агрегатор реализован как Next.js API Route (`/app/api/metadata/route.ts`) с внутренней архитектурой из трёх независимых модулей: `fetcher.ts` (HTTP-транспорт), `parser.ts` (извлечение данных из HTML), `cache.ts` (кэширование). Граница между модулями задаётся через интерфейс `LinkMetadata`; замена любого модуля (например, переход с in-memory кэша на Redis) не затрагивает остальные.

Транспортный слой реализован следующим образом. HTTP GET выполняется с заголовком `User-Agent: Mozilla/5.0 (compatible; EmbeddedCardsBot/1.0)` - самоидентификация агента требуется как этическая норма и как условие корректной обработки на серверах, настроенных на блокировку ботов без `User-Agent`. Заголовок `Accept-Language: ru,en;q=0.9` запрашивает локализованные метаданные там, где сервер поддерживает content negotiation. Таймаут 6 секунд управляется `AbortController`; выбор продиктован NFR T-04 (суммарный отклик ≤ 3 с): при таймауте 6 с агент завершает ожидание с запасом к клиентскому NFR с учётом сетевых задержек обоих хопов. Тело ответа считывается потоково с лимитом 2 МБ - при превышении потоковое чтение прерывается через `reader.cancel()`. Лимит выбран как верхняя граница типичного HTML-документа с полным контентом: страницы с SPA-бандлами часто превышают 5-10 МБ, однако метаданные и тело контента сосредоточены в первых 200-400 КБ; 2 МБ покрывают этот диапазон с запасом.

Парсинговый слой строится на жёстко заданной последовательности источников OG → JSON-LD → Twitter Cards → HTML meta → fallback реализована жёстко и не конфигурируется пользователем. Обоснование: OG покрывает медиа-ресурсы, агрегаторы и блог-платформы, составляющие более 80% бытового трафика ссылок; JSON-LD - новостные и коммерческие сайты, внедрившие разметку Schema.org; Twitter Cards - нишевые технические ресурсы, намеренно не заполняющие OG; meta-теги - корпоративные и устаревшие страницы. Fallback на `H1` + первый абзац закрывает случаи полного отсутствия метаданных - например, статические HTML-документы без шаблонов. Позволить пользователю конфигурировать приоритет означало бы перенести на него ответственность за решение, оптимальность которого определяется глобальным распределением форматов, а не индивидуальными предпочтениями.

На рисунке 13 представлена полная схема обработки запроса агрегации: от инициации на клиенте до ответа с DTO или graceful degradation.

Алгоритм извлечения тела страницы (`excerpt`, `wordCount`) реализует трёхфазный пайплайн. Фаза 1 - удаление шума: из дерева DOM удаляются `nav`, `header`, `footer`, `aside`, `script`, `style`, `noscript`, `iframe`, `svg`, `[aria-hidden="true"]`, а также элементы с классами-индикаторами рекламы и меню (`.nav`, `.menu`, `.sidebar`, `.ad`, `.cookie`, `#header`, `#footer`). Фаза 2 - поиск семантического контейнера: перебираются `article`, `[role="main"]`, `main`, `.post-content`, `.entry-content`, `.article-body`, `.story`, `#content`; первый найденный с текстом более 200 символов используется как корень для извлечения. Если ни один не найден - используется `body`. Фаза 3 - нормализация: текст извлекается через `.text`, нормализуются пробелы, подсчитываются токены (split по `\s+`), формируется excerpt из первых 600 символов. Порог 200 символов для контейнера и лимит 600 символов для excerpt выбраны эмпирически: меньший порог включает сайдбары и шапки, больший лимит перегружает карточку текстом. Результат извлечения кэшируется вместе с остальными полями DTO.

Слой кэша реализован как `Map<string, CacheEntry>`, который инициализируется один раз при загрузке модуля Node.js и живёт в памяти серверного процесса. Ключ - исходный URL (не canonical), поскольку клиент запрашивает именно тот URL, который ввёл пользователь; канонический URL вычисляется уже в парсере. Инвалидация реализована по стратегии «lazy expiry»: просроченная запись обнаруживается и удаляется в момент чтения (`cacheGet`), а не по таймеру. Это устраняет необходимость в фоновом планировщике и упрощает тестирование модуля. Ограничение: при перезапуске серверного процесса (деплой, crash) кэш теряется. Для продакшн-сценария требуется внешний KV (Redis, Upstash) с тем же TTL-интерфейсом - это зафиксировано в роадмапе как первоочередное улучшение.

Компонент `LinkCard` рендерится как `<a>` с `target="_blank" rel="noopener noreferrer"`. Порядок отображения контента: (1) favicon через Google S2 API + домен + время чтения (`wordCount / 200` мин); (2) заголовок; (3) excerpt - если есть, иначе description; (4) разделитель + источник метаданных (`source`) + иконка внешней ссылки. Изображение отображается как верхний баннер при наличии `image`; при ошибке загрузки `onError` скрывает блок без нарушения компоновки. `LinkCardSkeleton` - анимированная заглушка с геометрически совпадающей структурой - предотвращает cumulative layout shift: высота скелетона соответствует высоте карточки с данными.

Компонент `MDXPreview` реализует клиентский рендер MDX-разметки без полноценного компилятора. Это намеренное инженерное упрощение для MVP: полная компиляция MDX в браузере добавит ~150 КБ к клиентскому бандлу (сам `@mdx-js/mdx` вместе с зависимостями) и потребует eval или Function-конструктора, что создаёт поверхность для XSS при рендеринге пользовательского контента. Вместо этого `MDXPreview` разбивает текст регулярными выражениями на блоки (`<LinkCard ... />` - компонент, остальное - Markdown), обрабатывает базовое форматирование (`#`, `**`, `_`, `` ` ``) и рендерит React-дерево. Это достаточно для целевого сценария MVP; полноценный MDX-рендеринг зафиксирован как Should-улучшение следующей итерации.

Хук `useMetadata` управляет четырьмя состояниями жизненного цикла: `idle → loading → success | error`. Ответы кэшируются в `useRef(new Map())` на время сессии - повторный вызов с тем же URL не генерирует сетевой запрос. Хук не запускает запрос при пустом, невалидном или undefined URL: предикат проверки выполняется до `useEffect`, что исключает лишние HTTP-запросы на каждый keystroke при незавершённом вводе. Этот клиентский кэш дополняет серверный: при обновлении страницы клиентский кэш сбрасывается, но серверный сохраняется (до 24 ч), поэтому повторные запросы после перезагрузки возвращаются из серверного кэша за 2-4 мс.

Хук `useDocuments` реализует CRUD-операции над `Doc[]` в localStorage: создание, обновление (по ID через `map`), удаление, дублирование, переименование. `hydrated: boolean` - guard против рассинхронизации серверного HTML и клиентского состояния localStorage при SSR: рендер приложения блокируется до завершения гидрации. Экспорт через `URL.createObjectURL(new Blob([content], { type: 'text/plain' }))` формирует `.mdx`-файл, содержимое которого побайтово совпадает с редактируемым исходником (T-05).

Оба пути вставки карточки (Quick URL bar и автодополнение через `UrlSuggestion`) завершаются одним MDX-тегом: `<LinkCard url="..." />`. Унификация формата хранения принципиальна: если бы Quick URL bar формировал отличный синтаксис, экспортированный MDX содержал бы два варианта одного компонента, что нарушило бы обратную совместимость при переносе в другой фреймворк.

UMUX-Lite предпочтён полному SUS (10 вопросов) ввиду малой экспертной выборки и ограниченного времени сессии; валидность на выборках n < 10 при условии экспертного состава подтверждена в [15] и [32].

## 3.5 Экономическое обоснование проекта: TCO, амортизация, инфраструктурные затраты, NPV/ROI/срок окупаемости

Раздел построен по схеме «формула - подстановка - результат»: каждая ключевая метрика снабжена явным определением, после чего в неё подставляются фактические параметры разработанного MVP. Все расчёты выполнены на горизонте трёх лет (Год 0 - пилот, Годы 1-3 - продакшн), с фиксацией даты курсов и тарифов на 26 апреля 2026 г.

### 3.5.1 Обоснование исходных параметров расчёта

**Условная ставка времени пользователя.** Стоимость одного часа времени целевого пользователя (студент/аспирант/начинающий специалист) выражается формулой (1):

$$
C_{hour} = \frac{S_{month}\cdot (1 - n_{NDFL})}{H_{month}}
$$
(1)

где:  
$C_{hour}$ - стоимость часа труда (нетто), руб/ч;  
$S_{month}$ - среднемесячная начисленная заработная плата (брутто), руб/мес;  
$H_{month}$ - фонд рабочего времени, ч/мес;  
$n_{NDFL}$ - ставка НДФЛ, доля.

Подстановка по hh.ru [34] для начинающего ИТ-специалиста в Москве (брутто 90 000 руб/мес, $H_{month}=168$ ч, $n_{NDFL}=0{,}13$):

$$
C_{hour}^{IT} = \frac{90\,000\cdot(1-0{,}13)}{168} = \frac{78\,300}{168} \approx 466 \text{ руб/ч}
$$

Подстановка по Росстату [35] для базового сегмента (студент-аспирант, брутто 65 000 руб/мес):

$$
C_{hour}^{base} = \frac{65\,000\cdot 0{,}87}{168} = \frac{56\,550}{168} \approx 337 \text{ руб/ч}
$$

Таким образом, эмпирический диапазон $C_{hour}$ для целевой аудитории составляет 337-466 руб/ч. В дальнейших расчётах принято консервативное значение $C_{hour}=400$ руб/ч (медиана диапазона округлена в нижнюю сторону), что соответствует требованию занижения экономического эффекта при оценке инвестиционной привлекательности.

**Курс доллара США.** Все долларовые тарифы зарубежных сервисов (Vercel, Upstash, Cloudflare) переведены в рубли по курсу ЦБ РФ [31] на 26.04.2026: $K_{\$}=92{,}15$ руб/$. Пересчёт по формуле (2):

$$
Z_{rub} = Z_{usd} \cdot K_{\$}
$$
(2)

где:  
$Z_{rub}$ - затраты в рублях, руб;  
$Z_{usd}$ - затраты в долларах США, $;  
$K_{\$}$ - курс ЦБ РФ на дату расчёта, руб/$.

Подстановка для тарифа Vercel Pro ($Z_{usd}=20$ $/мес):

$$
Z_{rub}^{Vercel} = 20\cdot 92{,}15 = 1\,843 \text{ руб/мес}
$$

Подстановка для Upstash Redis Pay-as-you-go ($Z_{usd}=10$ $/мес):

$$
Z_{rub}^{Upstash} = 10\cdot 92{,}15 = 921{,}5 \approx 922 \text{ руб/мес}
$$

### 3.5.2 Разбивка инфраструктурных затрат TO-BE по статьям

Совокупные ежемесячные инфраструктурные затраты определяются как сумма по статьям тарификации Vercel [36], Upstash [37] и Cloudflare [38]. Расчёт по формуле (3):

$$
TCO_{infra,month} = \sum_{i=1}^{n} Z_{i,month}
$$
(3)

где:  
$TCO_{infra,month}$ - месячные инфраструктурные затраты, руб/мес;  
$Z_{i,month}$ - $i$-я статья затрат за месяц, руб/мес;  
$n$ - число статей затрат в инфраструктурной модели.

Годовые затраты получаются умножением на 12 (формула (4)):

$$
TCO_{infra,year} = 12 \cdot TCO_{infra,month}
$$
(4)

где:  
$TCO_{infra,year}$ - годовые инфраструктурные затраты, руб/год.

Подстановка для **пилотного режима** (Vercel Hobby + in-memory кэш + домен .ru):

### 3.5.3 Амортизация оборудования

$$
A_{month} = \frac{C_{asset}}{N_{life}}
$$
(5)

где:  
$A_{month}$ - месячная норма амортизации актива, руб/мес;  
$C_{asset}$ - первоначальная стоимость актива, руб;  
$N_{life}$ - срок полезного использования актива, мес.

Годовая норма амортизации - формула (6):

$$
A_{year} = 12 \cdot A_{month}
$$
(6)

где:  
$A_{year}$ - годовая норма амортизации актива, руб/год.

Часть амортизации, атрибутируемая проекту - формула (7):

$$
A_{proj} = A_{year} \cdot k_{use}
$$
(7)

где:  
$A_{proj}$ - проектная амортизация актива, руб/год;  
$k_{use}$ - доля времени использования актива в проекте (по дневнику практики).

Подстановка для основного актива - ноутбука разработчика (MacBook Air M2, $C_{asset}=130\,000$ руб, $k_{use}=0{,}6$):

$$

$$

$$
A_{year}^{laptop} = 12\cdot 3\,611 = 43\,332 \text{ руб/год}
$$

$$
A_{proj}^{laptop} = 43\,332 \cdot 0{,}6 \approx 25\,999 \text{ руб/год}
$$

Аналогичные подстановки для монитора ($C_{asset}=25\,000$ руб): $A_{proj}^{monitor}\approx 4\,997$ руб/год; для периферии ($C_{asset}=8\,000$ руб): $A_{proj}^{periph}\approx 1\,598$ руб/год.

Совокупная проектная амортизация - сумма по активам:

$$
A_{proj}^{total} = 25\,999 + 4\,997 + 1\,598 = 32\,594 \text{ руб/год} \approx 32{,}6 \text{ тыс.руб/год}
$$

Эта величина включается в годовые затраты $Z_t$ (формула (9)) на каждом периоде расчёта NPV.

### 3.5.4 Облачные сервисы и требования 152-ФЗ

При коммерциализации (запуск личного кабинета, синхронизация между устройствами) возникает обязанность хранить ПДн российских граждан на серверах в РФ согласно ст. 18 п. 5 ФЗ-152 [40]. Удорожание инфраструктуры от перехода на отечественное облако определяется формулой (8):

$$
\Delta TCO_{152} = TCO_{RU} - TCO_{nonRU}
$$
(8)

где:  
$\Delta TCO_{152}$ - премия за соответствие 152-ФЗ, руб/год;  
$TCO_{RU}$ - годовая стоимость владения при размещении в РФ, руб/год;  
$TCO_{nonRU}$ - годовая стоимость владения при нероссийском хостинге, руб/год.

Подстановка для целевой нагрузки 10 000 пользователей. По официальным калькуляторам Yandex Cloud [42] (Compute Cloud 4 vCPU + 8 GB RAM = 4 800 руб/мес, Object Storage 100 ГБ = 250 руб/мес, Managed Redis 1 ГБ = 1 450 руб/мес, исходящий трафик = 500 руб/мес) совокупная стоимость составляет:

$$
TCO_{RU,month} = 4\,800 + 250 + 1\,450 + 500 = 7\,000 \text{ руб/мес}
$$

$$
TCO_{RU,year} = 12 \cdot 7\,000 = 84\,000 \text{ руб/год}
$$

Тогда премия за соответствие 152-ФЗ:

$$
\Delta TCO_{152} = 84\,000 - 33\,180 = 50\,820 \text{ руб/год}
$$

В удельных затратах на пользователя:

$$
\Delta TCO_{152,user} = \frac{50\,820}{10\,000} = 5{,}08 \text{ руб/польз/год}
$$

Премия 5,08 руб/пользователь/год за легитимность хранения ПДн в РФ - на два порядка меньше любой типовой годовой подписки и не требует пересмотра тарифа Pro (290 руб/мес = 3 480 руб/год).

### 3.5.5 Потенциальные затраты на расширение и поддержку

### 3.5.6 Расчёт NPV, ROI и срока окупаемости

Выручка периода - сумма доходов B2C и B2B - формула (10):

$$
R_{t} = R_{B2C,t} + R_{B2B,t} = N_{B2C,t}\cdot p_{B2C} + N_{B2B,t}\cdot p_{B2B}
$$
(10)

где:  
$R_{t}$ - выручка периода $t$, тыс.руб;  
$N_{B2C,t}$ - число платящих B2C-пользователей;  
$p_{B2C}=3{,}48$ тыс.руб/год - годовая цена подписки Pro (290 руб/мес × 12);  
$N_{B2B,t}$ - число подразделений-подписчиков B2B;  
$p_{B2B}=50$ тыс.руб/год - стоимость B2B-лицензии на 100 пользователей.

Подстановка для Года 1 (100 B2C + 1 B2B):

$$
R_{1} = 100\cdot 3{,}48 + 1\cdot 50 = 348 + 50 = 398 \text{ тыс.руб}
$$

Подстановка для Года 2 (400 B2C + 3 B2B):

$$
R_{2} = 400\cdot 3{,}48 + 3\cdot 50 = 1\,392 + 150 = 1\,542 \text{ тыс.руб}
$$

Подстановка для Года 3 (800 B2C + 6 B2B):

$$
R_{3} = 800\cdot 3{,}48 + 6\cdot 50 = 2\,784 + 300 = 3\,084 \text{ тыс.руб}
$$

Чистый денежный поток - формула (11):

$$
CF_{t} = R_{t} - Z_{t}
$$
(11)

где:  
$CF_{t}$ - чистый денежный поток периода $t$, тыс.руб;  
$R_{t}$ - выручка периода $t$;  
$Z_{t}$ - совокупные затраты периода $t$.

Подстановка по годам:

$$
CF_{0} = 0 - 69{,}1 = -69{,}1; \quad CF_{1} = 398 - 501{,}4 = -103{,}4
$$

$$
CF_{2} = 1\,542 - 757{,}9 = +784{,}1; \quad CF_{3} = 3\,084 - 607{,}4 = +2\,476{,}6
$$

**Ставка дисконтирования.** Применена ставка $r=28\%$ годовых. Ключевая ставка ЦБ РФ на 26.04.2026 составляет $r_{CBR}=16\%$ [45]. Премия за индивидуальный риск стартап-проекта принята $r_{risk}=12$ п.п., что соответствует [33]:

$$
r = r_{CBR} + r_{risk} = 0{,}16 + 0{,}12 = 0{,}28
$$

Дисконт-факторы по годам:

$$
(1+r)^0 = 1{,}000; \quad (1+r)^1 = 1{,}280; \quad (1+r)^2 = 1{,}6384; \quad (1+r)^3 = 2{,}0972
$$

Приведённая стоимость потока периода - формула (12):

$$
PV_{t} = \frac{CF_{t}}{(1+r)^t}
$$
(12)

где:  
$PV_{t}$ - приведённая стоимость денежного потока периода $t$, тыс.руб;  
$r$ - ставка дисконтирования, доля;  
$t$ - порядковый номер периода, лет.

Подстановка:

$$
PV_{0} = \frac{-69{,}1}{1{,}000} = -69{,}1; \quad PV_{1} = \frac{-103{,}4}{1{,}280} = -80{,}8
$$

$$
PV_{2} = \frac{784{,}1}{1{,}6384} = +478{,}7; \quad PV_{3} = \frac{2\,476{,}6}{2{,}0972} = +1\,181{,}2
$$

Чистая приведённая стоимость проекта - формула (13):

$$
NPV = \sum_{t=0}^{T}\frac{CF_{t}}{(1+r)^t}
$$
(13)

где:  
$NPV$ - чистая приведённая стоимость, тыс.руб;  
$T=3$ - горизонт расчёта, лет.

Подстановка:

$$
NPV = -69{,}1 + (-80{,}8) + 478{,}7 + 1\,181{,}2 = +1\,510{,}0 \text{ тыс.руб}
$$

Положительное значение $NPV>0$ означает, что проект создаёт стоимость 1,51 млн руб. сверх требуемой нормы доходности 28% годовых.

Рентабельность инвестиций - формула (14):

$$
ROI = \frac{\sum_{t=0}^{T}R_{t} - \sum_{t=0}^{T}Z_{t}}{\sum_{t=0}^{T}Z_{t}}\cdot 100\%
$$
(14)

где:  
$ROI$ - рентабельность инвестиций (без учёта дисконтирования), %;  
$R_{t}$, $Z_{t}$ - выручка и затраты периода (формулы (10), (9)).

Подстановка:

$$
\sum R_{t} = 0 + 398 + 1\,542 + 3\,084 = 5\,024 \text{ тыс.руб}
$$

$$
\sum Z_{t} = 1\,935{,}8 \text{ тыс.руб (см. § 3.5.5)}
$$

$$
ROI = \frac{5\,024 - 1\,935{,}8}{1\,935{,}8}\cdot 100\% = \frac{3\,088{,}2}{1\,935{,}8}\cdot 100\% \approx 159{,}5\%
$$

Показатель ROI рассчитан **без дисконтирования** (суммарная прибыль к суммарным затратам за 3 года) и носит вспомогательный характер: он завышает привлекательность относительно NPV при коротком горизонте и малых стартовых затратах. Для оценки инвестиционной целесообразности в работе используются **NPV, PI и DPP** при $r=28\%$; $ROI_{avg}\approx 53\%$ годовых приведён только как ориентир сопоставления с $r$.

Индекс рентабельности - формула (15):

$$
PI = \frac{\sum_{t:\,PV_{t}>0}PV_{t}}{\sum_{t:\,PV_{t}<0}|PV_{t}|}
$$
(15)

где:  
$PI$ - индекс рентабельности инвестиций, отн.ед;  
числитель - сумма положительных дисконтированных потоков;  
знаменатель - сумма абсолютных значений отрицательных дисконтированных потоков.

Подстановка:

$$
PI = \frac{478{,}7 + 1\,181{,}2}{|-69{,}1| + |-80{,}8|} = \frac{1\,659{,}9}{149{,}9} \approx 11{,}07
$$

Значение $PI\approx 11$ означает, что каждый рубль инвестиций возвращает 11 рублей дисконтированного дохода - значительно выше нормативного порога $PI>1$.

Внутренняя норма доходности (IRR) определяется условием $NPV(r^*)=0$. Подбор $r^*$ выполнен численно: при $r=200\%$ получаем $NPV>0$, при $r=290\%$ - $NPV\approx 0$, следовательно $IRR\approx 290\%$.

$$
NPV(r=290\%) \approx 0 \Rightarrow IRR \approx 290\%
$$

Высокое значение IRR объясняется **низкими затратами нулевого года** ($Z_0=69{,}1$ тыс.руб, $CF_0=-69{,}1$) при отсутствии крупных капитальных вложений и сильном положительном потоке в год 3 ($CF_3=+2\,476{,}6$ тыс.руб). Это типично для проектов с минимальным стартовым cash-out; для принятия решения в работе опорным показателем принят **дисконтированный NPV** при $r=28\%$, а не IRR.

Дисконтированный срок окупаемости - формула (16):

$$
DPP = t^{*} + \frac{|NPV_{cum,t^{*}}|}{PV_{t^{*}+1}}
$$
(16)

где:  
$DPP$ - дисконтированный срок окупаемости, лет;  
$t^{*}$ - последний период с отрицательным кумулятивным NPV;  
$NPV_{cum,t^{*}}$ - кумулятивный NPV в конце периода $t^{*}$, тыс.руб;  
$PV_{t^{*}+1}$ - приведённая стоимость потока следующего периода, тыс.руб.

Кумулятивный NPV по годам:

$$
NPV_{cum,0} = -69{,}1; \quad NPV_{cum,1} = -69{,}1 - 80{,}8 = -149{,}9
$$

$$
NPV_{cum,2} = -149{,}9 + 478{,}7 = +328{,}8 \Rightarrow t^{*}=1
$$

Подстановка:

$$
DPP = 1 + \frac{149{,}9}{478{,}7} = 1 + 0{,}313 = 1{,}313 \text{ года} \approx 15{,}8 \text{ мес}
$$

С учётом сезонности B2C-продаж (всплеск в начале учебного года) практический срок окупаемости составляет 14-16 месяцев от начала продаж.

## 3.6 Ценностное предложение и обоснование коммерческой реализуемости проекта

Раздел оформляет коммерческую логику проекта от лица потенциального инвестора: что покупает клиент, какую ценность измеримо получает, как продукт сравнивается с альтернативами и какова дорожная карта возврата вложений.

### 3.6.1 Сегментация клиентов и количественная оценка рынка

Совокупный достижимый объём рынка (SOM) на горизонте 3 лет рассчитывается как доля от исходной целевой аудитории, переведённая в ARPU-эквивалент - формула (17):

$$
SOM_{seg} = N_{seg} \cdot c_{seg} \cdot ARPU_{seg}
$$
(17)

где:  
$SOM_{seg}$ - достижимый объём сегмента, тыс.руб/год;  
$N_{seg}$ - целевая аудитория сегмента, чел/орг;  
$c_{seg}$ - целевая конверсия в платящих, доля;  
$ARPU_{seg}$ - средний доход с клиента в сегменте, тыс.руб/год.

Подстановка для **B2C** (студенты, аспиранты, исследователи; $N_{B2C}=1{,}35$ млн чел [46], $c_{B2C}=0{,}07\%$ к Году 3, $ARPU_{B2C}=3{,}48$ тыс.руб):

$$
SOM_{B2C} = 1\,350\,000 \cdot 0{,}0007 \cdot 3{,}48 = 945 \cdot 3{,}48 \approx 3\,289 \text{ тыс.руб/год}
$$

Принятый в плане выручки целевой объём 800 платящих в Году 3 (см. таблицу 22) соответствует $c_{B2C}\approx 0{,}059\%$ - на 16% ниже расчётной целевой конверсии, что закрепляет консервативность прогноза.

Подстановка для **B2B-вузы** (700 вузов РФ, целевая конверсия 0,86% = 6 подразделений к Году 3, $ARPU_{B2B}=50$ тыс.руб):

$$
SOM_{B2B,vuz} = 700 \cdot 0{,}0086 \cdot 50 = 6{,}02 \cdot 50 = 301 \text{ тыс.руб/год}
$$

Подстановка для **B2B-нишевый** (5-10 тыс. техписов в РФ; в плане выручки этот сегмент учтён внутри B2C-подписок Pro, поскольку техпис покупает индивидуальную подписку):

$$
SOM_{B2B,techwriters} = 7\,500 \cdot 0{,}01 \cdot 3{,}48 \approx 261 \text{ тыс.руб/год}
$$

Совокупный SOM на Год 3:

$$
SOM_{total} = 3\,289 + 301 + 261 \approx 3\,851 \text{ тыс.руб/год}
$$

План выручки $R_{3}=3\,084$ тыс.руб использует 80% расчётного SOM, что оставляет запас прочности 20% к рыночному ограничению.

### 3.6.2 Решение и измеримая выгода для клиента

**Экономия времени пользователя.** Сокращение времени обработки одной ссылки с $t_{ASIS}=30$ с до $t_{TOBE}=5$ с (фактический медианный отклик API 1,24 с + ~3,76 с на проверку и подтверждение в UI; подтверждено в § 3.4) даёт экономию по формуле (18):

$$
\Delta T = N_{url}\cdot \frac{t_{ASIS} - t_{TOBE}}{3\,600}
$$
(18)

где:  
$\Delta T$ - экономия времени за период, ч;  
$N_{url}$ - число обработанных ссылок за период;  
$t_{ASIS}$, $t_{TOBE}$ - время одной операции до и после внедрения, с.

**Денежная оценка экономии** - формула (19):

$$
E_{rub} = \Delta T \cdot C_{hour}
$$
(19)

где:  
$E_{rub}$ - денежный эквивалент экономии, руб;  
$C_{hour}=400$ руб/ч (см. § 3.5.1).

Подстановка для **активного исследователя B2C** ($N_{url}=1\,500$ ссылок/год):

$$
\Delta T_{B2C} = 1\,500 \cdot \frac{30 - 5}{3\,600} = 1\,500 \cdot 0{,}00694 \approx 10{,}4 \text{ ч/год}
$$

$$
E_{rub}^{B2C} = 10{,}4 \cdot 400 = 4\,166 \text{ руб/год}
$$

Стоимость подписки Pro - 3 480 руб/год. Чистая выгода клиента:

$$
NB_{B2C} = E_{rub}^{B2C} - p_{B2C} = 4\,166 - 3\,480 = +686 \text{ руб/год}
$$

Подписка Pro окупается уже на исходной экономии времени; оставшаяся ценность (унифицированный формат, переносимость, Excerpt вместо OG-тизера, отсутствие vendor lock-in) идёт сверху как чистая прибыль клиента.

Подстановка для **подразделения вуза** ($N_{users}=100$ чел, средний $N_{url,user}=1\,500$ ссылок/год):

$$
N_{url,dept} = 100 \cdot 1\,500 = 150\,000 \text{ ссылок/год}
$$

$$
\Delta T_{B2B} = 150\,000 \cdot \frac{25}{3\,600} \approx 1\,042 \text{ ч/год}
$$

$$
E_{rub}^{B2B} = 1\,042 \cdot 400 = 416\,667 \text{ руб/год}
$$

Стоимость B2B-лицензии 50 000 руб/год. Чистая выгода подразделения:

$$
NB_{B2B} = 416\,667 - 50\,000 = +366\,667 \text{ руб/год}
$$

ROI клиента (со стороны подразделения вуза):

$$
ROI_{client} = \frac{NB_{B2B}}{p_{B2B}}\cdot 100\% = \frac{366\,667}{50\,000}\cdot 100\% \approx 733\%
$$

Иными словами: каждый рубль, потраченный подразделением на лицензию, возвращается в виде 8 рублей сэкономленного рабочего времени сотрудников на горизонте одного года - при этом неучтённой остаётся стоимость отказа от Notion Plus (отдельно см. § 3.6.3).

### 3.6.3 Стратегические преимущества: количественное сравнение TCO

Коэффициент ценового преимущества по сравнению с конкурентом - формула (20):

$$
K_{price} = \frac{TCO_{comp}}{TCO_{our}}
$$
(20)

где:  
$K_{price}$ - кратность преимущества (раз);  
$TCO_{comp}$ - годовая стоимость владения у конкурента, руб/год;  
$TCO_{our}$ - годовая стоимость владения проектируемой платформой, руб/год.

Подстановка для **B2C, тариф Free** ($TCO_{our}=0$ руб/год). Сравнение с Notion Plus ($TCO_{comp}=10$ $/мес × 12 × 92,15 ≈ 11\,058$ руб/год):

$$
K_{price}^{Notion-Plus} = \frac{11\,058}{0+200} = 55{,}3 \text{ (учитывая amortизацию домена 200 руб/год)}
$$

При покупке тарифа **Pro** ($TCO_{our}=3\,480$ руб/год):

$$
K_{price}^{Notion-Plus,Pro} = \frac{11\,058}{3\,480} \approx 3{,}18
$$

Подстановка для **B2B (100 пользователей)**. Notion Business: $20$ $/мес × 100 × 12 × 92{,}15 = 2\,211\,600$ руб/год. Наша лицензия - 50 000 руб/год:

$$
K_{price}^{Notion-Business,B2B} = \frac{2\,211\,600}{50\,000} \approx 44{,}2
$$

Абсолютная экономия для подразделения вуза при переходе с Notion Business на нашу платформу:

$$
\Delta TCO_{B2B} = 2\,211\,600 - 50\,000 = 2\,161\,600 \text{ руб/год}
$$

Эта величина в 4,3 раза превышает экономию рабочего времени сотрудников ($E_{rub}^{B2B}=416\,667$ руб/год) и сама по себе формирует основной коммерческий аргумент для университета.
